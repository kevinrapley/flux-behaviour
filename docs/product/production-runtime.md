# Production runtime

Flux Behaviour Pages serves a D1-backed API at `/api/collect`, `/api/auth/google/start`, `/api/auth/google/callback` and `/api/dashboard/researchops`.

The collector accepts only consented, metadata-only events for the `researchops` tenant from its configured origins. It stores a random, browser-held visitor ID and a browser-session ID; neither is an account identifier. The dashboard builds a plain-language journey narrative from controlled event metadata, never typed values or individual words.

Google sign-in requires protected Pages secrets `FLUX_AUTH_SECRET`, `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET`. The Google OAuth client must register `https://flux-behaviour.pages.dev/api/auth/google/callback` as its production redirect URI. A Google identity is attached to the pre-seeded Flux account by verified Google email and is stored separately from visitor and session analytics data.

The tracker creates persistent visitor and session identifiers only after a visitor grants consent; revoking consent clears both identifiers. Services can supply `data-flux-page`, `data-flux-key` and `data-flux-role` attributes for stable semantic page, control, field and form names. Type-first keys such as `tab.journal.analysis` and `field.analysis.code-retrieval` let the dashboard explain purpose without reading page content; page narratives require the literal controlled `page.` namespace. Otherwise, the tracker creates a neutral `auto.*` structural key, and legacy or URL-derived page keys remain generic. It never derives an analytics key from visible text, accessible labels, form values, names or IDs. Email, telephone, password and one-time-code inputs are excluded entirely from auto-capture and can be explicitly marked `data-flux-sensitive="true"`; sensitivity applies to every descendant control. Auth tokens are detected case-insensitively at any segment of a semantic key, and forms containing an auth-scoped descendant are sensitive. Every auth-scoped semantic interaction is rejected by both automatic and manual SDK/collector validation. The reserved `auth.otp` key is valid only for consent-gated requested, succeeded and failed lifecycle actions with the neutral service shape and no optional metadata, including code length. The dashboard groups controlled narrative events by their pseudonymous session, including whether the session belongs to a returning visitor, and aggregates only approved metadata: event counts, returning sessions, pre-input field dwell, active typing duration and rate, character counts, corrections, paste and shortcut categories, validation transitions, help disclosures, submit attempts and controlled semantic keys. Total focus time remains available but is never presented or scored as dwell once an input interaction occurred.

Journey narratives translate those controlled keys and actions into purpose-led sentences. A `field.blur` value length is rejected unless positive edit, paste or keypress evidence exists; actual visitor edits retain bounded character-count and dwell metadata. Typing speed uses standard gross WPM from printable non-shortcut keystrokes divided by five over active typing time, so prefilled words and Ctrl/Cmd/Alt shortcuts cannot inflate the current interaction's rate. When a publisher supplies the complete optional writing bundle, Flux identifies the declared `en-GB` baseline and reports possible spelling and grammar issue counts plus casing proportions for the current field content. Those counts must be produced on-device after consent; the collector rejects partial bundles and never receives entered text, words, fragments or suggestions. Controlled page loads, non-sensitive form submissions, validation errors and allow-listed authentication outcomes use action-specific language, while legacy and structural fallback events remain visibly generic rather than being given guessed meaning.

For each session, Flux calculates the 20 indicators used by the demo model and five composites. The calculation is bounded to observed, consented metadata; an unsupported indicator remains neutral and no value is inferred from identity, text or a protected field. A sufficiently high possible-issue ratio can add a small bounded cognitive-load and frustration signal as a prompt to inspect the service journey, but never changes literacy, intelligence or professionalism scores. These are service-friction heuristics for human review, not classifications or judgements of an individual. The promoted mapping and linguistic signal have not yet completed representative golden-corpus, accessibility or fairness validation, so GAP-008 and GAP-012 prohibit automated, high-stakes or person-level use.

The dashboard defaults to the last 30 days and supports 7-day, 90-day and all-time views. Visitors, returning visitors, sessions, interactions, completion and friction are calculated cumulatively for the selected period, with like-for-like comparison data and daily audience trends. The primary view is aggregate service evidence and does not expose raw pseudonymous visitor identifiers.

The realtime panel separately reports active sessions and accepted interactions over the last 5 and 30 minutes, a complete per-minute series and ingestion freshness derived from server-controlled `accepted_at_ms`. It contains no live identity, location or journey snapshot. See `docs/product/realtime-analytics.md`.

Flux now stores a versioned publisher service model for services, transactions, tasks, steps, questions and fields, including question complexity, required/optional field status, semantic bindings, configured outcomes and action-specific key events. The collector freezes the resolved hierarchy and exact model version beside each event. Generic submits remain interaction evidence but do not count as completion: dashboard completion, outcome cohorts and journey status require a publisher-configured success outcome. See `docs/product/publisher-service-model.md`.

The event report ranks event classes and actions, configured key events, and semantic pages, forms and controls with event-active journey reach, daily trends and like-for-like volume movement. The service-performance report rolls exact-version event context up through transactions, tasks, steps, questions and fields, showing unique journeys, entry, exit, configured success, explicit friction and average elapsed time. Both reports use accessible, horizontally contained tables and expose no visitor or session identifiers. See `docs/product/event-entity-reports.md`.

The dashboard also compares privacy-preserving journey cohorts for visit maturity, outcome paths and supported interaction patterns. Named cohorts are suppressed until at least five journeys share the cohort. Visit-maturity and outcome cohorts cover the selected period; interaction patterns use up to the 250 most recent complete journey histories and disclose when the sample is limited. Cohorts describe journeys rather than people, neutral evidence produces no dominant pattern, and education, personality, protected-characteristic and browser-fingerprinting inferences are prohibited. See `docs/product/cohort-analytics.md`.

Recent journeys use a bounded event window for responsive overview rendering, but each displayed journey offers an authenticated complete-history request. That request reads the session's full, tenant-scoped event sequence in chronological order and recomputes its indicators from that full sequence.

Live collection is not release approval. DPIA, accessibility evidence, retention/deletion policy, production rate limiting and incident controls remain release blockers.
