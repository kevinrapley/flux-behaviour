# Inclusivity testing plan baseline

Status: baseline plan. This is not completed research evidence.

## Purpose

Flux Behaviour must be tested for fair interpretation across different users, assistive technologies, confidence levels, stress levels and interaction styles.

The key question is not whether a signal exists. The key question is whether the product interprets that signal safely and usefully.

## Testing objectives

1. Check whether Flux interpretations match user-reported experience.
2. Identify where signals risk pathologising neurodivergent, disabled or low-confidence users.
3. Validate whether barrier and strength interpretations are both visible to reviewers.
4. Check whether consent copy is understandable and trustworthy.
5. Check whether dashboards avoid deficit framing.

## Proposed participant coverage

Recruit a mixed cohort covering, where appropriate and consented:

- ADHD;
- autism;
- dyslexia;
- anxiety or stress-related access needs;
- screen reader users;
- keyboard-only users;
- speech input users;
- low digital confidence;
- mobile-only access;
- neurotypical users.

Do not force disclosure. Use self-description and access needs where possible.

## Method

1. Run representative service journeys with consented metadata capture.
2. Capture participant self-report on confidence, trust, effort and perceived barriers.
3. Compare product interpretations with participant accounts.
4. Separate service friction from user behaviour.
5. Review false positives and false negatives by cohort where sample size and consent allow.
6. Update thresholds, copy, guidance and harm register entries when issues are found.

## Outputs

- Inclusivity validation log.
- Threshold review notes.
- Copy changes for consent and dashboard explanations.
- Harm register updates.
- Release recommendation.

## Release rule

No live deployment should proceed until inclusivity risks have been reviewed and residual risks are explicitly accepted by the product owner and governance lead.
