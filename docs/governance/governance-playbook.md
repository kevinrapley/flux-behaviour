# Governance playbook baseline

Status: baseline operating guidance. This is not a live operational playbook.

## Purpose

This playbook explains how reviewers should interpret Flux Behaviour outputs safely.

The product should help teams ask better questions about service friction. It must not make unsupported claims about individual users.

## Interpretation principles

1. Treat signals as prompts for investigation.
2. Prefer trend-level evidence over individual-session judgement.
3. Always consider benign and positive explanations.
4. Combine telemetry with user research, service data and accessibility evidence.
5. Record reviewer decisions, overrides and rationale.
6. Escalate harm risks rather than suppressing inconvenient findings.

## Validation panel

A validation panel should show:

- precision;
- recall;
- false-positive rate;
- false-negative rate where known;
- review coverage;
- cohort-level caveats;
- model or ruleset version.

Minimum expectation: reviewers should be able to see whether a trigger is well evidenced before acting on it.

## Trust and adaptation panel

Trust indicators must be interpreted carefully.

Repeated checking might indicate low trust, high stakes, careful behaviour, previous negative experience, cognitive load or a sensible verification strategy.

Assisted completion must not be framed as lower ability. It may indicate good service design, effective help, assistive technology use or user expertise.

## Flow and transition views

Flow views can help teams see repeated movement into friction states.

They must not be used to label people as risky, deficient or non-compliant.

## Composite views

Composite scores should show uncertainty, definitions and source signals.

Flat charts may mean balanced support. Spiky charts may mean uneven service support. They should not be treated as user capability profiles.

## Reviewer actions

Reviewers may:

- mark a trigger as valid;
- mark a trigger as false positive;
- add context;
- request user research follow-up;
- request accessibility review;
- update a service improvement backlog;
- escalate a harm-register issue.

## Release rule

No dashboard interpretation should be released without explanatory copy, reviewer guidance, version information and a route to challenge or correct interpretation.
