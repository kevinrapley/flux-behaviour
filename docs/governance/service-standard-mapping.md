# Service Standard mapping baseline

Status: baseline mapping for review. This is not a completed service assessment.

Flux Behaviour supports service improvement evidence. It does not replace user research, accessibility testing, analytics governance, performance reporting or service assessment.

| Service Standard point | Flux Behaviour contribution | Required safeguards |
| --- | --- | --- |
| 2. Solve a whole problem for users | Highlights journey points where users pause, backtrack, repeat errors or seek help. | Combine with user research and service data. Do not treat telemetry as the whole evidence base. |
| 5. Make sure everyone can use the service | Identifies friction patterns that may suggest barriers for some users, including neurodivergent users. | Do not infer disability or protected characteristics. Validate with inclusive research and accessibility testing. |
| 10. Define what success looks like and publish performance data | Provides service-improvement indicators for completion support, recovery behaviour, confidence and friction. | Publish only aggregate, non-identifying performance evidence with clear caveats. |
| 12. Make sure users succeed first time | Highlights repeated errors, backtracking, field revisits, help-seeking and abandon-risk signals. | Use as diagnostic evidence for improving the service, not for judging individual users. |
| 14. Operate a reliable service | Supports versioned schemas, audit records, release gates and trend-level operational review. | Maintain schema versioning, retention policy, incident handling and production monitoring. |

## Assessment boundary

Flux Behaviour can support a service team preparing evidence for assessment, but it is not itself evidence that a service meets the Service Standard.

The product must remain accountable to service teams, governance reviewers and users.
